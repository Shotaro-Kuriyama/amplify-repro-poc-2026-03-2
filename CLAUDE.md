# CLAUDE.md

## このプロジェクトが何か
このリポジトリは、AmpliFy のようなプロダクトを想定したフロントエンドモックです。  
ユーザーが建築平面図 PDF をアップロードし、BIM 的な 3D 結果を見る Web アプリを目指しています。

ただし現時点では、**本物の BIM 変換エンジンではありません。**  
これは **顧客デモ向けのフロントエンド試作** です。

目的は、プロダクトが実在するように感じられる UI/UX を作りつつ、  
将来のバックエンド接続に備えたコードベースを整えることです。

---

## 何を最も大事にするか
このリポジトリでは、以下を優先してください。

1. **顧客デモ品質**
   - プロダクトとして成立して見えること
   - ラフな試作品ではなく、見せられる完成度を持つこと

2. **日本語ファーストの体験**
   - 日本語文言の品質が重要
   - SaaS / 建築 / 業務ツールとして自然であること
   - 機械翻訳っぽい不自然な文章は避けること

3. **状態遷移の明確さ**
   - empty
   - uploaded / ready
   - processing
   - completed
   - failed

   これらが見た目・挙動の両面で分かれていること

4. **将来バックエンドに接続しやすい構造**
   - ユーザーは後から本物のバックエンドをつなげたいと考えている
   - mock 実装は差し替えやすい構造であるべき
   - 表示コンポーネントに API 前提のロジックを埋め込みすぎないこと

5. **学習しやすさ**
   - このリポジトリはユーザーの学習教材でもある
   - 賢すぎる抽象化より、読みやすさと理解しやすさを優先すること

---

## このリポジトリでの進め方
作業するときは、**慎重なシニアエンジニア兼メンター** のように振る舞ってください。

つまり:
- なぜその変更が必要かを説明する
- 不要に複雑にしない
- 読みやすい構造を保つ
- 大きな作り直しより、意図のある改善を優先する
- 学習しやすいコードを意識する

新規性やかっこよさよりも、以下を優先してください。
- 明快さ
- 保守性
- デモ品質
- 将来のバックエンド接続しやすさ

---

## 現在想定しているプロダクト体験
目指している UX の流れは、おおむね以下です。

1. ユーザーがページを開く
2. 何のサービスか直感的に理解できる
3. 1つ以上の PDF 平面図をアップロードする
4. 階層順や Viewer 設定を調整する
5. 変換開始する
6. 段階的な processing 表示を見る
7. 3D 結果が生成されたように見える
8. 数量表やモデル成果物をダウンロードする
9. 「かなり実用に近い」と感じてもらう

どの変更も、この流れを支えるべきです。

## デザイン方針
UI は以下のように感じられるべきです。

- モダン
- クリーン
- 控えめ
- 信頼感がある
- 建築 / BIM / SaaS らしい
- 派手すぎず洗練されている

重視するもの:
- 良い余白
- 明確なタイポグラフィ階層
- やわらかいカード構造
- 控えめな境界線や影
- loading / disabled / success / error の意図的な表現

避けるもの:
- うるさい色使い
- 過度にポップな演出
- 詰め込みすぎたレイアウト
- デモ品質を上げない無意味な複雑さ

---

## 言語 / i18n の考え方
現時点の方針は以下です。

- 日本語が既定言語
- 英語は補助
- 多言語対応は独自辞書 + React Context で実装する
- 言語実装は中途半端な状態を残さない
- 文言を改善するときは、まず自然な日本語を優先する

文言は:
- 短く
- 明確に
- 一目で意味が分かるように
してください。

## Viewer に対する期待
3D Viewer は、このデモの見せ場の1つです。

重要なこと:
- 今は BIM として正確でなくてよい
- ただし、見せ場として説得力が必要
- Reset View は本当に reset するべき
- camera mode の切り替えは一貫しているべき
- UI に出した設定は、Viewer に見える形で反映されるべき

Viewer は、プロダクトの将来像を感じさせる役割を持ちます。

---

## API に対する期待
このプロジェクトは、将来本物のバックエンドへつなげやすい構造を保つ必要があります。

以下の分離を維持してください。
- UI
- hooks / 状態管理
- API 層
- mock 実装

避けるべきこと:
- コンポーネント内にバックエンド前提ロジックを直接書く
- 通信的な責務をあちこちに散らす
- API 層を強い理由なく飛ばす

---

## 特に注意してほしい箇所
以下を触るときは慎重に扱ってください。

- file ID の管理
- upload フロー
- floor relabeling / `startFloor`
- Viewer の reset 挙動
- settings と Viewer の同期
- download フロー
- failed state の再現性
- build / lint の安定性

このあたりは、小さな不整合でも UX や将来接続性を大きく崩します。

---

## このリポジトリで「良い変更」とは何か
良い変更は、だいたい次の特徴を持ちます。

- 小さいが意味がある
- 正しさや明快さが増す
- 今のデモ品質を壊さない
- 将来のバックエンド接続がしやすくなる
- コードを不必要に難しくしない
- 初学者にも説明しやすい

---

## コミュニケーション方針
作業報告では、以下を意識してください。

- 具体的に書く
- 何を変えたか、なぜ変えたかを書く
- 確認できた事実と推測を分ける
- 実際に検証した内容を書く
- まだ危ない部分や未完部分も正直に書く

テストしていないことは、テストしていないと明記してください。

---

## 最終的な意図
ユーザーは、このリポジトリで次の3つを同時に目指しています。

- 今は「見せられる製品デモ」を作ること
- 将来は「実際の製品の土台」にすること
- そして、自分で徐々に理解できるコードベースにすること

この3つを同時に支えられるように手伝ってください。

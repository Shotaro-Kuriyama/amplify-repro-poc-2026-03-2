## 今回の目的

API / パイプラインの最小確認は進んでいる前提で、次は以下を確認したいです。

1. **real モードでブラウザ UI が自然に動くか**
2. **mock モードでも UI が破綻しないか**
3. **Viewer の挙動に不具合があるか**
4. 不具合がある場合、**原因を切り分けて、必要最小限の修正**を行う

特に、以前ブラウザ上で

* PDF アップロード後にいきなり 3D 建物が出るように見える
* Viewer がズームし続ける / カメラ挙動が不自然に見える
  と感じたため、この点を重点的に確認してください。

## 最初に守ってほしいこと

* 大きな設計変更はしない
* API 境界を壊さない
* `src/lib/api/client.ts` を API 入口として扱う
* Route Handler と pipeline-runner の責務を混ぜない
* 変更は小さく安全に行う
* まず **再現確認と原因切り分け** を優先し、すぐに大きく直し始めない
* build / lint / 型の安定性を壊さない

## 今回やってほしい作業

### 1. real モードのブラウザ UI 確認

`npm run dev` で real モードを起動し、ブラウザで実際に確認してください。

見てほしい流れ:

* 初期表示
* PDF アップロード
* ready 状態
* create job 後の processing 表示
* completed 時の Viewer 表示
* failed 時の表示
* artifact / quantities 取得導線
* reset view の挙動
* settings 変更時の反映

特に確認してほしい点:

* completed 前に Viewer が不自然に 3D 完成表示になっていないか
* processing → completed の切り替わりが自然か
* completed 後の初期カメラ位置が不自然でないか
* 何も操作していないのにズームし続けるように見えないか
* reset view が本当に reset しているか

### 2. mock モードのブラウザ UI 確認

`NEXT_PUBLIC_API_MODE=mock npm run dev` でも確認してください。

見てほしい点:

* createAmplifyJob を `"queued"` に揃えたことで UI が壊れていないか
* processing 表示が自然か
* completed / failed の遷移が問題ないか
* real と mock の UI 体験に不要な差がないか

### 3. Viewer 不具合の切り分け

以下の可能性を意識して切り分けてください。

候補例:

* `Viewer3D.tsx` の bounds 計算
* `fitCameraToBounds()` の再実行タイミング
* `useEffect` / `useMemo` による再計算ループ
* `resetViewTrigger` や `controls.saveState()` / `reset()` 周り
* `pipelineModel` / `sceneBounds` の変化によりカメラが過剰に再調整されていないか
* `BuildingModel.tsx` 側のモデルサイズや座標変換の影響
* completed 前後の状態切り替えで Canvas が再 mount される影響

やってほしいこと:

* まず再現するかどうかを確認
* 再現するなら、どの条件で起きるか整理
* 原因候補を絞る
* 必要なら最小限のログやコメントで確認
* 修正する場合は、**最小限の安全な修正**だけ行う

### 4. 修正方針

もし Viewer 不具合が見つかった場合は、

* まず原因の説明
* その原因に対して最小限の修正
* なぜその修正でよいのか
  を明確にしてください。

避けてほしいこと:

* なんとなく広範囲に Viewer を書き換える
* 不具合原因が確定していないのに大規模リファクタする
* 共通化や美化を優先する

### 5. 作業後の確認

修正した場合は、少なくとも

* `npm run lint`
* `npm run build`
  を確認してください。

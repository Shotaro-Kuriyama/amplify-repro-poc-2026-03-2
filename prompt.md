## 今回の目的

この repo はすでに
**front + 擬似 backend + Python pipeline + 最小 IFC 生成**
まで入った最小縦切り PoC です。

次は、これを壊さずに

1. 小さい不整合を直す
2. real モードの最小パイプラインが本当に動くか確認する
   ところまで進めたいです。

## 最初に守ってほしいこと

* 大きなリファクタはしない
* UI を壊さない
* API 境界を壊さない
* `src/lib/api/client.ts` を API 入口として扱う
* Route Handler と pipeline-runner の責務を混ぜない
* 変更は小さく安全に行う
* build / lint / 型の安定性を最優先する

## 今回やってほしい作業

### 1. エラーコード整合の修正

まず、`src/lib/api/errors.ts` と、Route Handlers / pipeline-runner 側で実際に使っているエラーコードの不一致を整理してください。

確認対象の例:

* `src/lib/api/errors.ts`
* `src/lib/server/pipeline-runner.ts`
* `src/app/api/jobs/[jobId]/artifacts/[format]/route.ts`
* 他にも実際にエラーコードを返している Route Handler があれば確認

やってほしいこと:

* 現在 real 側で実際に使っているエラーコードを洗い出す
* `errors.ts` の定義と整合させる
* 無理に大規模な抽象化はしない
* 今の実装に素直に合わせる
* 必要ならコメントも最小限修正する

### 2. `src/types/pipeline.ts` の古いコメント修正

このファイルには、現状の実装とズレた古いコメントが残っている可能性があります。

やってほしいこと:

* 「まだ未接続」「Phase 8A で実装予定」のような古い説明を、現状に合わせて修正する
* ただし型定義そのものを不要にいじらない
* あくまで**理解を妨げる古い説明の修正**を主目的にする

### 3. mock / real の `createAmplifyJob` 契約差の整理

`mock.ts` の `createAmplifyJob` が `"processing"` を返し、real 側が `"queued"` を返している差があるはずです。

やってほしいこと:

* 現在の UI がこの差にどう依存しているか確認する
* もし安全に揃えられるなら、mock 側も real 側に寄せることを検討する
* ただし UI や既存挙動を壊すくらいなら無理に変えない
* 変更する場合は、なぜ安全だと判断したかも報告してください

### 4. 最小実動作確認

上の修正後、可能なら以下を確認してください。

* `npm run lint`
* `npm run build`
* real モードでの最小動作確認

  * upload
  * create job
  * polling
  * completed / failed
  * `structured_json` / `ifc` の取得

可能なら、`docs/phase8a-quickstart.md` に沿って最小確認をしてください。
ただし、実行環境の制約でできないことがあれば、**できたこと / できていないことを明確に分けて報告**してください。

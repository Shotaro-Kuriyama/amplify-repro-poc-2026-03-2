## 修正したい不具合

### 1. 変換開始後、進捗が十分に見える前にすぐ completed 扱いになって 3D モデルが表示されてしまう
現状は「処理が早い」というより、フロントエンドのポーリング設計のせいで、
途中の進捗表示をほとんど見ずに completed を拾ってしまっている可能性が高いです。

特に以下を疑っています。
- `src/hooks/useAmplifyJob.ts`
- 初回ポーリングが遅い
- `createAmplifyJob()` 後、最初の `getAmplifyJob()` が 1.5 秒後なので、その前に completed になってしまう
- その結果、15% / 40% / 65% / 85% の途中表示をほぼ見ないまま completed 表示に飛ぶ

### 2. ブラウザをフルスクリーンにすると、3D モデルがズームし続けるように見え、ページ全体の縦スクロール量が異常に大きくなる
これはカメラ制御というより、レイアウトの高さ設計の問題である可能性が高いです。

特に以下を疑っています。
- `src/app/page.tsx`
- 右側の viewer 領域が `h-full` ベース
- さらに completed 時に `PipelineResultCard` が下に追加される
- 親 flex / 子 flex に `min-h-0` が不足していて、高さがうまく閉じず、ページ全体が縦に伸びる
- その結果、Canvas の見え方もおかしくなり、「ズームしている」ように見える

# 修正1: useAmplifyJob のポーリング改善

対象:
- `src/hooks/useAmplifyJob.ts`

やってほしいこと:
1. 初回ポーリングを遅延させず、`createAmplifyJob()` 成功直後に **即時で 1 回 `getAmplifyJob()` を呼ぶ** ようにする
2. その後の通常ポーリング間隔も見直し、現状の `1500ms` より短くする
   - まずは `400ms` 前後を目安にしてよい
3. completed を拾った場合でも、UI 上の切り替えがあまりにも唐突なら、
   - 必要に応じてごく短い表示安定化（例: 数百 ms 程度）を入れてよい
   - ただし過剰な演出は不要
4. 既存の API 契約は壊さない
5. 型安全性を維持する
6. 既存の `reset()` や polling cleanup の設計を壊さない

修正方針:
- なるべく局所的に直す
- バックエンド API ではなく、まずはフロントのポーリング設計で改善する
- 「実処理が速いこと」と「進捗 UI が見えないこと」を混同しないこと

期待する結果:
- 変換開始後、少なくとも processing 状態が自然に見える
- completed への遷移が今より不自然でなくなる
- 途中段階の進捗 UI を観察しやすくなる

---

# 修正2: page.tsx の viewer レイアウト改善

対象:
- `src/app/page.tsx`

やってほしいこと:
1. 右側の main / viewer 領域の flex 高さ設計を見直す
2. `min-h-0` が必要な親子要素に適切に追加する
3. Viewer を `h-full` 依存ではなく、**flex レイアウトの中で自然に伸縮する構造**に寄せる
4. completed 時に `PipelineResultCard` が追加されても、
   - viewer 領域が暴走してページ全体の高さを異常に押し広げないようにする
5. フルスクリーン相当の大きな表示でも、
   - 3D viewer の表示領域が安定し
   - ページ全体の縦スクロールが不自然に伸びないようにする
6. デザインや構成は大きく壊さず、現状の UI をなるべく維持する

修正方針:
- 外側コンテナ、aside、main、viewer wrapper の責務を整理する
- `flex-1` と `min-h-0` を適切に使う
- 「viewer の高さ」と「結果カードの高さ」が競合しない構造にする
- mobile / desktop 両方で破綻しないこと

期待する結果:
- completed 後も viewer の高さが安定する
- ページ全体の縦スクロールが異常に増えない
- フルスクリーン表示でも「ズームし続けるように見える」症状が改善する

## 3. フェーズ進行の全体像

```
Phase 5.5  実機確認・デモ運用の仕上げ
Phase 6    API契約の固定
Phase 7    バックエンドの最小接続
Phase 7.5  実ファイル保持・永続化・ジョブ基盤の整理
Phase 8    PDF→BIM の技術検証
  ├─ 8A  ルールベース検証
  ├─ 8B  ML補助検証
  └─ 8C  精度評価と運用設計
```

全体方針は、いきなり本物の PDF→BIM を完成させることではなく、段階的にモックから PoC、PoC から技術検証へ進めることです。
各フェーズは、前のフェーズの成果を壊さず、小さく安全に積み上げる形で進めます。

## 4. 各フェーズの目的

### Phase 5.5 — 実機確認・デモ運用の仕上げ

**目的**: UI/UX やデモ体験を整え、実機で触って違和感の少ない状態にする。

主な内容:
- レスポンシブ対応
- 操作導線の管理
- ダウンロードフローの磨き込み
- エラー時の見せ方の整理
- デモ用途として破綻しない振る舞いの調整

**判断基準**: 「顧客や関係者の前で動かしても、見た目・操作感の面で恥ずかしくない」状態になっていること。

### Phase 6 — API契約の固定

**目的**: フロントとバックエンドの契約を固定し、request / response / error shape をぶらさないようにする。

主な内容:
- `types.ts` に 6 API の request/response 型を定義
- `schemas.ts` に Zod runtime validation スキーマ
- `errors.ts` にエラーコード・ApiError クラス
- UI 状態（idle/uploading/ready）と API 状態（queued/processing/completed/failed）の責務分離
- `docs/api-contract.md` に人間向け契約ドキュメント

補足:
- validation は段階的な導入でよい
- 特に JSON body を中心に契約を明確化し、upload や    params は必要に応じて補強する

**判断基準**: 「backend を誰が実装しても、contract.md を見れば何を返すべきか分かる」状態。

### Phase 7 — バックエンドの最小接続

**目的**: mock 直呼びから、HTTP 境界を持つ擬似 real API に移行する。upload → job 作成 → poll → download の流れを HTTP 込みで成立させる。

主な内容:
- `real.ts`: fetch ベースの API クライアント
- 6 本の Route Handlers（`/api/*`）
- in-memory store（時間ベースの job 進捗計算）
- 1.5 秒固定間隔 polling
- `NEXT_PUBLIC_API_MODE` による mock/real 切替
- JSON body を中心とした request validation（Zod）

補足:
- ここではまだ、本物の PDF→BIM 本体は実装しない
- あくまで「モックだが、HTTP 境界込みで動く」状態を作る

**判断基準**: 「client.ts の向き先を変えるだけで外部 backend に繋がる」構造になっていること。

### Phase 7.5 — 実ファイル保持・永続化・ジョブ基盤の整理

**目的**: Phase 8 に入る前の土台整理。in-memory の限界を超え、実際のファイルとジョブ状態を永続化する。

やるべきこと:
- **実ファイル保持**: アップロードされた PDF を一時ストレージに保存する（ローカル `/tmp` or S3）
- **永続化**: job 状態を DB（SQLite / PostgreSQL）に保存し、サーバー再起動でも保持する
- **job 状態管理**: ステータス遷移の整合性を DB レベルで保証する
- **worker 分離の準備**: 重い処理を Route Handler の外に出せる設計にする

注意点:
- 現在の in-memory 構成はローカルデモには向くが、永続性や安定したジョブ管理には向かない
- serverless / Vercel 系の実行環境では、実行時間、メモリ、ファイルシステム、インスタンス継続性に制約があるため、重い処理や長いジョブをそのまま載せる前提にはしない
- ここで大規模基盤を一気に作る必要はない。まずは最小の永続化方針を決めることが重要
- **判断基準**: 「Phase 8 で PDF を実際に受け取り、処理し、結果を返すための最低限の土台が整理されている」こと。

### Phase 8 — PDF→BIM の技術検証

**目的**: PDF→BIM 変換の中身を、段階的に技術検証する。
いきなり本番品質や大規模運用を目指すのではなく、まずは動く技術パイプラインを作ることを重視する。

#### Phase 8A — ルールベース検証

目的:
最初の一歩。Python スクリプトで PDF から壁・開口部を検出し、IFC を生成する。

主な内容:
- PDF から線分・テキスト・画像情報を抽出
- 壁・開口部・部屋候補をルールベースで推定
- 最小限の構造化データを生成
- まずは IFC の最小生成 を目標にする

補足:
- 精度は高くなくてよい
- 重要なのは、入力 → 抽出 → 判定 → 出力 の流れが一本つながること

#### Phase 8B — ML 補助検証

目的:
ルールベースでは拾いきれない部分を、ML を補助的に使って改善できるかを検証する。

主な内容:
- 壁・開口部・部屋領域などの推定補助
- 既存モデルの転用可能性の検証
- ルールベースと ML の役割分担の整理

補足:
- この段階で重い学習基盤を作る必要はない
- まずは「ML を入れるとどこが改善するか」を見る

#### Phase 8C — 精度評価と運用設計

目的:
技術検証の結果を評価し、どこまで実用に近づけるかを整理する。

主な内容:
- 精度評価指標の設計
- 処理時間の把握
- 図面の種類や制約条件の整理
- 運用上の前提条件の明文化

補足:
- ここでは「高精度化」だけでなく、「どの条件なら使えるか」を整理することも重要

## 6. この repo 単体では厳しいこと

今の Next.js / Route Handler / Vercel 中心の構成だけでは、以下を完結させるのは現実的でない。

| 項目 | 理由 |
|---|---|
| 重いモデル学習 | GPU や長時間計算が必要になりやすい |
| 重い長時間推論 | 実行時間・メモリ・安定性の制約を受けやすい |
| 本格的な PDF→BIM パイプラインの本番運用 | ファイル I/O・中間成果物管理・再試行制御が重い |
| 大きなファイルの安定ジョブ処理 | 永続化・キュー・ワーカー分離が必要になる |

補足:
- 補助的な ML 推論や軽い技術検証は十分可能
- ただし、重い学習や本番品質の長時間処理は別基盤が必要になる

これらが必要になった段階で、下記の理想アーキテクチャへの移行を検討してください。

## 7. 将来の理想アーキテクチャ

将来的には以下のような責務分離が自然です。

```
┌─────────────────────────────────────────────────┐
│ Next.js (Frontend)                              │
│ UI / upload / progress / download               │
│ ↕ fetch (REST API)                              │
├─────────────────────────────────────────────────┤
│ API Server (FastAPI / Next.js Route Handlers)   │
│ 認証 / リクエスト受付 / ジョブ管理                   │
│ ↕                                               │
├──────────────┬──────────────────────────────────┤
│ Storage      │ DB / Queue                       │
│ PDF 原本      │ job 状態                          │
│ 中間成果物     │ retry / step 管理                 │
│ 生成物(IFC等)  │ PostgreSQL + Redis / SQS         │
│ S3 / GCS     │                                  │
├──────────────┴──────────────────────────────────┤
│ Worker / Python Service                         │
│ PDF 解析 / ルールベース処理 / ML 推論 / IFC 生成.    │
│ GPU 環境（ML 使用時）                             │
└─────────────────────────────────────────────────┘
```

役割の考え方:
- Next.js は UI と導線
- API Server は受付と job 管理
- Storage はファイル置き場
- DB / Queue は状態管理と再試行制御
- Worker / Python Service は重い計算処理

この分離は一気にやる必要はなく、Phase 7.5 と Phase 8 を通じて段階的に育てていきます。

## 8. 開発上の判断基準

迷ったときは以下の順で判断してください。

1. **build / lint が壊れないこと** — これが最優先
2. **既存のデモ体験を壊さないこと** — 顧客に見せられる状態を常に保つ
3. **API 契約を守ること** — Phase 6 で固定した契約をむやみに変えない
4. **小さく安全に進めること** — 大規模な作り直しより、意味のある小さな改善
5. **読みやすさ** — 賢すぎる抽象化より、理解しやすいコード
6. **将来の差し替えやすさ** — ベタ書きでも、境界が明確なら OK

補足:
- 過剰な抽象化は不要
- ただし、mock / real / worker などの境界は明確に保つ
- 「今のための実装」と「将来差し替える場所」を混ぜないことが重要

- 本格的な認証・権限管理
- 大規模な DB / Queue 基盤の導入

- 重い ML 学習基盤の整備
- RVT / DWG など複数形式の本格対応
- WebSocket 導入
- UI の全面リニューアル
- デプロイ基盤の大規模見直し

補足:
- 永続化やファイル保持の検討は Phase 7.5 で行う
- ただし、ここでやるのは必要最小限の整理であり、大規模構築ではない

## 10. 次の最小アクション

Phase 7.5 前半の土台整理が完了した今、次に取り組むべきは **Phase 8A** です。
（Phase 7.5 後半の DB 永続化は、Phase 8A の処理要件に応じて判断する）

### Phase 7.5 前半で完了したこと

1. **アップロード PDF の実ファイル保存** ✓
   - `FileStorage` インターフェースと `LocalFileStorage` を導入
   - upload Route Handler がファイル実体を `data/uploads/` に保存するようになった

2. **永続化境界の整理** ✓（インターフェースのみ。DB 実装は未着手）
   - `FileRepository` / `JobRepository` / `LeadRepository` インターフェースを定義
   - 現在は InMemory 実装、将来 SQLite / PostgreSQL に差し替え可能

3. **Worker 分離の設計** ✓
   - 各 Route Handler に責務境界コメントを追加
   - 「何が Route Handler に残り、何が Worker に移るか」が明確になった

4. **Phase 8A の最小入力・最小出力の定義** ✓（一部未接続あり）
   - `src/types/pipeline.ts` に型を定義
   - `PipelineInput` → `ExtractedFloorData` → `PipelineOutput` の流れを型で表現
   - ただし `floorLabel` は現在フロントエンド専用で、サーバー側 API に未反映（Phase 8A で接続予定）

### Phase 8A 最小縦切りで完了したこと

1. **PipelineInput の組み立て** ✓
   - `src/lib/server/pipeline.ts` に `buildPipelineInput()` を追加
   - `StoredJob` + `StoredFile` から `PipelineInput` を構築
   - `floorLabel` は暫定で `1F, 2F...` と自動採番（正式接続は後続で対応）

2. **Python 最小 PDF 処理** ✓
   - `scripts/pipeline/extract_pdf.py` を追加（PyMuPDF 使用）
   - ページサイズ取得、テキストブロックからの部屋名候補抽出
   - `PipelineOutput` の shape に沿った JSON を返却

3. **内部用 API エンドポイント** ✓
   - `POST /api/internal/pipeline/run` を追加
   - 既存 6 API 契約には影響しない独立したエンドポイント
   - Node から Python を子プロセスとして呼び出す

4. **動作確認ガイド** ✓
   - `docs/phase8a-quickstart.md` に手順を記載

### Phase 8A 後半（線分抽出・壁候補推定）で完了したこと

1. **drawing 情報からの線分抽出** ✓
   - `page.get_drawings()` から line / rect を抽出
   - curve / bezier / arc は今回は対象外
   - 座標は mm 単位に統一

2. **最小ルールベースでの壁候補推定** ✓
   - 長さ 50mm 以上、水平/垂直から 5° 以内の線分を壁候補として採用
   - `ExtractedWall[]` を実データで返却
   - `stats.totalWalls` を実データで更新
   - thickness は stroke width または矩形短辺から推定（暫定）
   - confidence は固定値 0.5（暫定）

3. **精度は暫定**
   - 高精度化は Phase 8B 以降で行う
   - 斜め壁、曲線壁、開口部はまだ未対応

### Phase 8A 継続（壁候補の重複整理・マージ）で完了したこと

1. **壁候補の正規化** ✓
   - 水平線: startX <= endX に統一
   - 垂直線: startY <= endY に統一

2. **重複除去** ✓
   - 始点・終点がともに 2mm 以内の wall を同一とみなして除去
   - thickness が大きい方を残す

3. **同一直線上マージ** ✓
   - 水平/垂直線を対象に、同一直線上 (3mm 以内) で近接 (5mm 以内) する壁を 1 本に統合
   - 効果: walls_only PDF で 37→13 本、line_only PDF で 20→13 本に整理

4. **開口部・IFC 生成にはまだ進んでいない**
   - 壁候補の品質安定が先。開口部は壁が安定してから着手する

### Phase 8A 継続（thickness 推定の改善）で完了したこと

1. **rect 由来 thickness の信頼性判定** ✓
   - rect 短辺 <= 20mm → 壁厚として信頼 (source = "rect")
   - rect 短辺 > 20mm → 部屋寸法とみなし fallback (大きな矩形の誤った壁厚 150mm/160mm を排除)

2. **thickness source の追跡と優先順位** ✓
   - 内部処理で source ("rect" > "nearby_rect" > "line_stroke" > "fallback") を追跡
   - dedup/merge 時に信頼度の高い thickness を優先して採用

3. **fallback thickness の補完** ✓
   - 同一ページの rect 由来 thickness の中央値で fallback wall を補完
   - fallback デフォルト値を 150mm → 5mm (paper mm) に修正

4. **まだ本格的な壁厚推定ではない**
   - 平行線ペアからの壁厚推定は未実装
   - 図面スケールを考慮した実寸変換は未実装

### Phase 8A 継続（開口部候補の暫定推定）で完了したこと

1. **壁間ギャップベースの opening 推定** ✓
   - マージ後の壁リストから同一直線上の壁間ギャップを検出
   - ギャップ幅 8mm〜40mm (paper mm) を開口部候補として採用
   - `ExtractedOpening[]` を実データで返却
   - `stats.totalOpenings` を実データで更新

2. **type の暫定分類** ✓
   - gap >= 14mm → "door" (1:50 で 700mm 実寸。一般的なドア幅)
   - gap < 14mm → "unknown" (窓と断定するには情報不足)

3. **精度は暫定**
   - 壁間ギャップのみで判定。円弧ドア記号や窓枠認識は未対応
   - height は壁厚ベースの仮値
   - confidence は固定値 0.4（壁候補より低い）
   - PDF の描画方法によっては開口部が 0 件のままの場合もある

4. **IFC 生成にはまだ進んでいない**
   - まずは壁 + 開口部の候補データを安定させてから着手する

### 次のステップ（Phase 8B に向けて）

1. **開口部推定の精度向上**
   - 円弧パターンからのドア検出
   - 窓枠形状の認識
   - 斜め壁からの opening 推定

2. **floorLabel の正式接続**
   - `createAmplifyJob` の request 拡張を検討
   - フロントからサーバーへ階数情報を渡す

4. **永続化の実体を入れる（必要に応じて）**
   - Phase 8A の処理が数十秒以上かかる場合、SQLite 導入を検討
   - 現時点では数十 ms で完了するため、in-memory のままで十分

### やらなくてよいこと（Phase 8A 現時点）

- 本格的な ML モデルの選定
- GPU 前提の基盤整備
- 大規模キューシステム導入
- 本番運用前提の全自動化
- 高精度な開口部検出（円弧ドア記号・窓枠認識はまだ Phase 8B 以降）
- IFC 本生成（構造化 JSON が先）
